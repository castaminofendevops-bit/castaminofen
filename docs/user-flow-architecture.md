# User Flow Architecture — Castaminofen

هدف این سند طراحی جریان‌های کاربری است تا تجربه‌ای شفاف، کم‌دردسر و استاندارد برای محصولی هم‌سطح اینستاگرام، یوتیوب و Castbox فراهم کند.

---

## 1. جریان کلی تجربه کاربر (User Journey)

### 1.1. گروه‌های اصلی کاربران

- کاربر شنونده (Listener)
- کاربر سازنده محتوا / پادکستر (Creator)
- مدیر / Admin

### 1.2. اصول طراحی

- مسیرها باید قابل‌پیش‌بینی و بدون بن‌بست باشند.
- هر صفحه باید تنها یک هدف اصلی داشته باشد.
- اقدام اولیه باید واضح، بزرگ و در دسترس باشد.
- فیدها و پروفایل‌ها باید یکی‌به‌یکی به محتوا و اقدام بعدی راهنمایی کنند.
- کمترین تلاش لازم برای کشف و ایجاد محتوا، پیگیری و تبدیل وجود داشته باشد.
- هر تعامل باید بازخورد فوری دهد.
- تمام حالت‌ها (loading, empty, offline, error) باید تعریف شوند.
- نباید اقدامات پنهان یا زیرمنوهای پیچیده ایجاد شود.

## 2. جریان‌های کلیدی

### 2.1. Feed اصلی / کاوش

#### User Story

به‌عنوان یک شنونده، می‌خواهم فید شخصی‌سازی شده ببینم تا سریعاً محتوای جدید یا پیشنهادی را پیدا کنم.

#### User Goal

کشف سریع، شروع گوش دادن یا دنبال کردن سازنده و بازگشت به محتوا.

#### Entry Points

- صفحه اصلی وب یا اپ
- تب "کاوش" یا "Discover"
- اعلان‌های داخلی

#### Navigation Path

1. صفحه اصلی فید باز می‌شود
2. نوار جستجو و نوار دسته‌بندی در بالا
3. ریل‌های "برای شما"، "پربازدید"، "از دنبال‌شده‌ها" و "جدید"
4. کاربر یک کارت محتوا را می‌زند
5. وارد صفحه جزئیات یا پخشر می‌شود

#### Primary Actions

- انتخاب کارت محتوا
- آغاز پخش فوری
- دنبال کردن سازنده
- ذخیره/علاقه‌مندی کردن محتوا

#### Secondary Actions

- مشاهده پروفایل سازنده
- دسترسی به جستجو
- اشتراک‌گذاری محتوا
- بازگشت به فید

#### Success States

- محتوا باز می‌شود و کاربر می‌فهمد چه اتفاقی افتاده
- در صورت دنبال کردن، تعداد دنبال‌کننده و وضعیت CTA به‌روز می‌شود
- اگر محتوا به لیست "بعداً گوش بده" اضافه شود، وضعیت واضح شود

#### Failure States

- بارگذاری فید شکست می‌خورد: پیام خطا همراه دکمه "تلاش مجدد"
- اگر کارت نمایش داده نشود: جایگزین "محتوا یافت نشد" و CTA جستجو

#### Loading States

- اسکلتی برای ریل‌ها
- placeholder کارت‌ها
- loader در دکمه‌ها و CTAها

#### Empty States

- زمانی که کاربر چیزی دنبال نکرده است: پیشنهاد سازنده‌ها
- زمانی که فید شخصی هنوز داده ندارد: "برای شروع، چند سازنده را دنبال کنید"

#### Offline States

- نشان "آفلاین" در بالای صفحه
- محدودیت به محتواهای ذخیره‌شده یا خطای واضح بدون بن‌بست

#### Permission Requests

- اجازه ارسال نوتیفیکیشن برای پیشنهادها
- اجازه دسترسی به رسانه (اگر ضبط صدا یا ویدیو داخل اپ نیاز بود)

#### Confirmation Dialogs

- نیازی به تایید برای مشاهده محتوا نیست
- تایید برای پاک کردن محتوای ذخیره شده یا لغو دنبال‌کردن گروهی

#### Recovery Paths

- دکمه "تلاش مجدد" برای بارگذاری مجدد
- لینک به جستجو یا دسته‌بندی‌های محبوب

#### Exit Points

- بازگشت به صفحه قبلی
- ورود به صفحه جزئیات محتوا یا پروفایل سازنده
- خروج کامل از اپ

#### سوالات هر صفحه

- چرا اینجا هستم؟ برای پیدا کردن محتوا.
- اقدام اصلی چیست؟ انتخاب و پخش محتوا.
- بعدش چه می‌شود؟ وارد صفحه جزئیات یا پخش می‌شوم.
- اگر شکست بخورد؟ پیام خطا و تلاش مجدد.
- چگونه بازیابی کنم؟ دکمه "تلاش مجدد" یا بازگشت به کاوش ساده.

#### Navigation Diagram

- Home / Discover → Content Card → Content Detail / Player
- Home / Discover → Creator Profile
- Home / Discover → Search

#### State Diagram

- Loaded → Empty / Data
- Loading → Success / Error
- Offline → Cached / Retry

### 2.2. صفحه پروفایل کاربر / Creator Profile

#### User Story

به‌عنوان یک کاربر، می‌خواهم پروفایل سازنده را ببینم تا بتوانم او را دنبال کنم، محتوایش را مرور کنم و درباره او اطلاعات بیشتری کسب کنم.

#### User Goal

تصمیم بگیرم سازنده را دنبال کنم یا محتوای مرتبط را ببینم.

#### Entry Points

- کلیک روی نام یا آواتار سازنده در فید یا جزئیات محتوا
- جستجو برای نام سازنده
- لینک مستقیم / share

#### Navigation Path

1. کاربر وارد پروفایل می‌شود
2. بخش خلاصه و آمار در بالا
3. ریل محتواهای اخیر
4. CTAهای Follow و مشاهده جزئیات بیشتر

#### Primary Actions

- دنبال کردن سازنده
- مشاهده محتوا
- بازدید از وب‌سایت یا social links سازنده (اختیاری)

#### Secondary Actions

- ارسال گزارش یا بلاک سازنده
- مشاهده دسته‌بندی‌ها
- مشاهده استوری/پیشنهاد

#### Success States

- وضعیت دنبال شدن به‌سرعت تغییر می‌کند
- اگر کاربر خودش سازنده نیست، CTA تبدیل به فرم درخواست دیده شود

#### Failure States

- بارگذاری پروفایل شکست می‌خورد: پیام خطا و دکمه "بازگشت به کاوش"
- اگر سازنده وجود نداشته باشد: صفحه 404 اختصاصی با پیشنهاد جستجو

#### Loading States

- اسکلتی برای هدر و لیست محتوا
- بارگذاری isFollowing

#### Empty States

- اگر سازنده هنوز محتوا ندارد: پیام "سازنده هنوز محتوایی منتشر نکرده"
- اگر کاربر سازنده نیست و هنوز درخواست نداده: CTA درخواست سازنده

#### Offline States

- پیام آفلاین و نسخهٔ کش‌شده اگر باشد
- دکمه refresh غیرفعال تا اتصال برقرار شود

#### Permission Requests

- هیچ درخواست مجوزی در این صفحه نباید ناگهانی ظاهر شود

#### Confirmation Dialogs

- برای دنبال کردن/لغو دنبال کردن نیاز به تایید فوری نیست؛ تنها تغییر وضعیت

#### Recovery Paths

- دکمه بازگشت
- لینک به کاوش یا جستجو

#### Exit Points

- بازگشت به فید
- رفتن به صفحه جزئیات محتوا
- بازگشت به نتایج جستجو

#### سوالات هر صفحه

- چرا اینجا هستم؟ برای شناختن سازنده و دنبال کردن یا بررسی محتوا.
- اقدام اصلی چیست؟ دنبال کردن یا دسترسی به محتوا.
- بعدش چه می‌شود؟ فید محتوای سازنده نمایش داده می‌شود یا user وارد محتوا می‌شود.
- اگر شکست بخورد؟ خطای واضح و لینک بازگشت.
- چگونه بازیابی کنم؟ دکمه retry یا بازگشت.

#### Navigation Diagram

- Creator Card → Creator Profile → Content Detail
- Creator Profile → Follow / Request Creator → Confirmation

#### State Diagram

- Loading → Profile Loaded / Empty / Error
- Follow Toggle → Following / Not Following
- Request Creator → Pending / Submitted

### 2.3. تبدیل شنونده به سازنده (Creator Onboarding)

#### User Story

به‌عنوان یک شنونده، می‌خواهم سریع و بدون پیچیدگی درخواست تبدیل به سازنده بدهم تا بتوانم محتوا بسازم.

#### User Goal

فرم درخواست را پر کنم، مدارک ارسال شود، سپس منتظر تایید باشم.

#### Entry Points

- CTA روی پروفایل کاربر
- لینک "تبدیل به سازنده" در منو
- onboarding prompt در صفحه شخصی

#### Navigation Path

1. کاربر از پروفایل یا منو وارد می‌شود
2. فرم درخواست باز می‌شود
3. فیلدهای کلیدی تکمیل می‌شود
4. درخواست ارسال می‌شود
5. کاربر به صفحه وضعیت درخواست یا پنل سازنده هدایت می‌شود

#### Primary Actions

- پر کردن نام، بیو، لینک نمونه کار
- آپلود مدرک هویتی یا نمونه کار
- ارسال درخواست

#### Secondary Actions

- مشاهده شرایط و قوانین
- کنسل کردن
- بازگشت

#### Success States

- نمایش پیام موفقیت و خلاصه وضعیت
- دکمه "بازگشت به پروفایل" یا "رفتن به پنل سازنده" در دسترس

#### Failure States

- خطا در آپلود فایل
- خطا در ارسال فرم
- فیلدهای الزامی پرنشده

#### Loading States

- spinner روی دکمه ارسال
- progress indicator فایل
- disable دکمه‌های دیگر

#### Empty States

- فرم اولیه بعد از ورود به صفحه
- اگر پیش‌تر درخواست زده شده: "درخواست شما در وضعیت بررسی است"

#### Offline States

- پیام "اتصال قطع است" و disable دکمه ارسال
- امکان تکمیل فرم بومی، ارسال پس از برگشت اتصال

#### Permission Requests

- اجازه استفاده از دوربین/گالری فقط اگر ضبط یا اسکن کارت داخل اپ باشد

#### Confirmation Dialogs

- قبل از خروج از فرم ناتمام: "آیا مطمئنید که می‌خواهید بدون ارسال خارج شوید؟"
- بعد از ارسال موفق: "درخواست شما ثبت شد"

#### Recovery Paths

- دکمه تلاش مجدد ارسال
- اطلاع از فیلدهای ناقص
- امکان ذخیره موقت فرم

#### Exit Points

- بازگشت به صفحه پروفایل
- رفتن به صفحه وضعیت درخواست
- ورود به صفحه اصلی

#### سوالات هر صفحه

- چرا اینجا هستم؟ برای درخواست تبدیل به سازنده.
- اقدام اصلی چیست؟ ارسال درخواست.
- بعدش چه می‌شود؟ درخواست ثبت و وضعیت نمایش داده می‌شود.
- اگر شکست بخورد؟ خطای مشخص و پیشنهاد اصلاح یا تلاش مجدد.
- چگونه بازیابی کنم؟ اصلاح فیلد، تلاش مجدد، یا خروج با هشدار.

#### Navigation Diagram

- CTA → Request Form → Submission → Confirmation / Status
- Request Form → Cancel / Back

#### State Diagram

- Draft Form → Submitting → Submitted / Error
- Submitted → Pending / Approved / Rejected

### 2.4. تولید محتوا / Creator Panel

#### User Story

به‌عنوان یک سازنده، می‌خواهم سریع محتوا بسازم، اپیزود اضافه کنم و منتشر کنم تا دسترسی شنونده‌ها را افزایش دهم.

#### User Goal

محتوا را بدون پیچیدگی ایجاد و منتشر کنم، و وضعیتش را مدیریت کنم.

#### Entry Points

- صفحه `creator` در منوی اصلی
- CTA "پنل سازنده" بعد از تایید شدن
- دکمه ساخت محتوا در پروفایل

#### Navigation Path

1. وارد پنل سازنده می‌شوم
2. لیست محتوا و status را می‌بینم
3. روی "ایجاد محتوا" یا یک محتوا کلیک می‌کنم
4. صفحه جزئیات محتوا یا اضافه کردن اپیزود باز می‌شود
5. انتشار یا بروزرسانی را انجام می‌دهم

#### Primary Actions

- ایجاد محتوا
- افزودن اپیزود
- انتشار محتوا

#### Secondary Actions

- ویرایش عنوان/توضیحات
- مشاهده آمار محتوا
- مشاهده یا حذف محتوا

#### Success States

- محتوای جدید با پیام موفقیت ایجاد می‌شود
- پس از انتشار، وضعیت `PUBLISHED` نمایش داده می‌شود

#### Failure States

- خطا در ثبت محتوا یا انتشار
- عدم دسترسی به پنل برای کاربر غیرسازنده

#### Loading States

- loader روی دکمه‌ها
- placeholder لیست محتوا

#### Empty States

- پیام "هنوز محتوایی ندارید" همراه CTA ایجاد

#### Offline States

- پیغام قطع اینترنت
- disable اقدامات جدید تا اتصال برگردد

#### Permission Requests

- برای آپلود فایل رسانه ممکن است به دسترسی فایل یا دوربین نیاز باشد

#### Confirmation Dialogs

- قبل از ترک صفحه محتوای در حال ایجاد، پیام هشدار
- تایید قبل از حذف محتوا

#### Recovery Paths

- دکمه ذخیره پیش‌نویس
- دکمه تلاش مجدد انتشار

#### Exit Points

- بازگشت به پنل
- رفتن به صفحه جزئیات محتوا
- ترک صفحه با هشدار در صورت تغییرات ذخیره نشده

#### سوالات هر صفحه

- چرا اینجا هستم؟ برای ساخت یا مدیریت محتوا.
- اقدام اصلی چیست؟ ایجاد/انتشار محتوا.
- بعدش چه می‌شود؟ محتوا در دسترس شنونده‌ها قرار می‌گیرد.
- اگر شکست بخورد؟ خطا و امکان تلاش مجدد.
- چگونه بازیابی کنم؟ ذخیره پیش‌نویس یا اصلاح ورودی.

#### Navigation Diagram

- Creator Panel → New Content → Add Episodes → Publish
- Creator Panel → Content Detail → Edit / Publish

#### State Diagram

- Empty → Draft Content → Published / Error
- Content List → Item Selected → Edit / Publish

### 2.5. سیستم دنبال کردن و تعامل اجتماعی

#### User Story

به‌عنوان یک شنونده، می‌خواهم سازنده‌ها را دنبال کنم تا فید من مرتبط‌تر و تازه‌تر باشد.

#### User Goal

فید شخصی‌تر و ارتباط نزدیک‌تر با سازنده‌ها داشته باشم.

#### Entry Points

- دکمه دنبال کردن روی کارت محتوا
- پروفایل سازنده
- دکمه‌های CTA زیر پخشگر

#### Navigation Path

1. کاربر دنبال کردن را می‌زند
2. UI بلافاصله به حالت دنبال شده تغییر می‌کند
3. فید به محتوای سازنده اضافه می‌شود

#### Primary Actions

- دنبال کردن/لغو دنبال کردن
- مشاهده فید دنبال‌شده‌ها

#### Secondary Actions

- ذخیره محتوا
- اشتراک‌گذاری محتوا

#### Success States

- وضعیت دکمه فوراً به "دنبال‌شده" تغییر می‌کند
- محتواهای جدید سازنده در بخش "از دنبال‌شده‌ها" نمایش داده می‌شوند

#### Failure States

- خطا در شبکه: پیام "اتصال برقرار نیست" و revert موقت
- خطای سرور: پیام خطا و امکان تلاش مجدد

#### Loading States

- indicator کوچک روی دکمه

#### Empty States

- "شما هنوز کسی را دنبال نکرده‌اید" با CTA دنبال کردن سازنده

#### Offline States

- دکمه دنبال کردن غیرفعال یا در صف انتظار قرار گیرد
- پیام offline

#### Permission Requests

- بدون نیاز خاص

#### Confirmation Dialogs

- برای لغو دنبال کردن نیازی به تایید ندارد مگر در حالت bulk

#### Recovery Paths

- تلاش مجدد بعد از خطا
- retry در فید

#### Exit Points

- بازگشت به فید
- ورود به پروفایل سازنده

#### سوالات هر صفحه

- چرا اینجا هستم؟ برای دنبال کردن سازنده‌ها.
- اقدام اصلی چیست؟ دنبال کردن.
- بعدش چه می‌شود؟ فید شخصی‌سازی می‌شود.
- اگر شکست بخورد؟ پیام خطا و retry.
- چگونه بازیابی کنم؟ دوباره کلیک یا بررسی اتصال.

#### Navigation Diagram

- Content Card / Profile → Follow → Feed Updates

#### State Diagram

- Not Following → Following / Error

## 3. پیاده‌سازی مورد نیاز در این repo

### 3.1. اجزای موجود

- صفحه اصلی `apps/web/src/app/(main)/page.tsx`
- پنل سازنده `apps/web/src/app/(main)/creator/page.tsx`
- صفحه پروفایل `apps/web/src/app/(main)/profile/[id]/page.tsx`
- فرم درخواست سازنده `apps/web/src/components/profile/ConvertToCreatorModal.tsx`

### 3.2. گلوگاه‌های معماری

- endpointهای `GET /creators/:id` و `/creators/:id/contents` باید قطعی باشند.
- صفحه‌ای برای وضعیت درخواست سازنده نیاز داریم (`pending`, `approved`, `rejected`).
- جریان Admin برای تایید درخواست‌ها هنوز ساخته نشده است.
- فید شخصی‌سازی‌شده و دیتای دنبال‌شده هنوز نیاز به مدل و مسیر دارد.

---

## 4. پیشنهاد اولویت اجرا

1. صفحه پروفایل کامل با CTA مشخص و درخواست سازنده
2. فرم درخواست سازنده با حالت pending و بازخورد فوری
3. Admin review panel برای تایید یا رد درخواست‌ها
4. فید دنبال‌شده‌ها و توصیه‌های شخصی‌شده
5. مدیریت محتوای سازنده و انتشار

---

## 5. پیشنهاد عملیات بعدی

- مستند کردن کامل صفحه Admin approval و مدل وضعیت requests.
- فاز بعدی: افزودن صفحه `creator/status` و endpoint `GET /creators/request`.
- سپس: جریان فید دنبال‌شده و محتوای پیشنهادی.
